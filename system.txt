GANTABYA SAHAYAK — SYSTEM OVERVIEW
===================================

Purpose
-------
Gantabya Sahayak is a Nepal tourism WebGIS application. It helps a traveller
find nearby places, analyse trekking routes, search hospitality services,
locate emergency facilities, and plan trips from personal constraints.

How to run
----------
1. Run start.bat or start.ps1 from the project root.
2. Frontend: http://localhost:3000/index.html
3. Backend:  http://localhost:8000
4. API docs: http://localhost:8000/docs

Architecture
------------
Frontend
- Plain HTML, CSS and JavaScript in frontend/.
- MapLibre GL JS renders maps and layers.
- Turf.js performs browser-side distance, buffers, nearest point and route
  overview calculations.
- The primary tool workspace is frontend/pages/tools.html. It changes tool
  behavior through the URL parameter: ?tool=nearest, trek, buffer, emergency,
  or planner.
- Shared utilities are in frontend/js/app.js.

Backend
- FastAPI application: backend/main.py.
- Serves GeoJSON, trip-planning APIs and live mapping lookups.
- Static data is available under http://localhost:8000/data/<file>.
- CORS is enabled for the frontend development server.

Five user tools
---------------
1. Nearest Location Finder
   - Filters mapped tourism locations by province, district and type.
   - Uses browser location permission to find the closest matching point.
   - Shows route guidance and a detailed selected-place brief.

2. Trekking Route Analyzer
   - Filters trekking routes by province and district.
   - Requests OpenStreetMap hiking relations when available.
   - Uses detailed offline trail geometry when live hiking geometry is absent.
   - Shows distance, duration, elevation, stage guide, permits, highlights and
     safety/packing information.

3. Hotel & Restaurant Buffer
   - Uses a user-selected radius in kilometres.
   - Searches OpenStreetMap for nearby hotels, guest houses, restaurants,
     cafés and fast-food locations.
   - Shows a buffer polygon, nearest-first results, marker details, services,
     contact data, indicative price tier and route guidance.

4. Emergency Services Finder
   - Searches live nearby hospitals, clinics/health posts and police offices.
   - Uses OpenStreetMap services and ranks leading candidates with OSRM road
     distance when available.
   - Displays an emergency detail card with route, distance, travel estimate,
     address, operator, hours, accessibility and contact details.

5. Smart Trip Planner
   - Collects available days, budget, transport, comfort, pace, group size and
     ticked interests.
   - Displays destinations specific to those interests.
   - Each destination opens a detailed plan with recommended accommodation,
     seasonal guidance, activities, travel estimate, cost/resource breakdown,
     preparation requirements and return-to-list control.

External live-data services
---------------------------
- OpenStreetMap Overpass API: mapped trekking paths, hospitality and emergency
  points of interest.
- OSRM public routing service: road-route geometry and route distance.
- Open-Meteo: weather endpoint used by the shared weather widget.

Privacy and limitations
-----------------------
- Browser location permission is required for nearest, buffer, emergency and
  location-based planner workflows.
- Live OSM/OSRM searches use the user's selected/current coordinates only after
  the user initiates and confirms the relevant live search.
- Public map data can be incomplete or out of date. Route, cost, availability,
  opening hours, permits and emergency information must be confirmed locally.
- Price levels are indicative where map data does not publish actual prices.

Main API routes
---------------
GET  /api/health
GET  /api/provinces, /api/districts, /api/local-levels
GET  /api/trekking, /api/trekking-waypoints
GET  /api/trekking/{route_name}/geometry
GET  /api/hospitality/nearby?lat=&lng=&radius_km=&type=
GET  /api/emergency/nearby?lat=&lng=&radius_km=&type=
GET  /api/emergency/nearest?lat=&lng=&type=&limit=
POST /api/trip/plan
GET  /api/weather?lat=&lng=
GET  /api/stats

Maintenance notes
-----------------
- Do not commit API keys; current live services do not require an API key.
- OpenStreetMap queries are cached in backend memory to reduce public-service
  load. Restarting the backend clears this cache.
- The local GeoJSON files remain fallbacks for offline/demo use.
